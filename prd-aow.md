

## **1\. Структура базы данных (Database Schema)**

### **Таблица event\_types**

| Поле | Тип | Описание |
| :---- | :---- | :---- |
| id | UUID | PK. |
| name | String | "Свадьба", "День Рождения" и т.д. |

### **Таблица event\_requests**

| Поле | Тип | Описание |
| :---- | :---- | :---- |
| id | UUID | PK. |
| name | String | Имя клиента (обязательное). |
| phone | String | Номер телефона (для быстрой связи). |
| event\_type\_id | Foreign Key | Связь с event\_types.id. |
| start\_date | Timestamp | Дата и время начала праздника. |
| end\_date | Timestamp | Дата и время окончания. |
| guests | Int | Число гостей. |
| budget | Int | Согласованный бюджет. |
| status | Enum | new, in\_progress, confirmed, completed, cancelled. |
| address | string | адрес мероприятия |
| comment | Text | Основной блок для деталей (пожелания, особые условия). |

## **2\. API Роуты (Backend)**

* **GET /api/event-types** — Загрузка списка типов для выпадающего списка (Select) в форме создания.  
* **POST /api/requests** — **Главный роут:** Создание новой заявки админом вручную.  
* **GET /api/requests** — Получение всех заявок (с JOIN event\_types) для отображения сетки.  
* **PATCH /api/requests/:id** — Обновление данных (если клиент перезвонил и поменял время/детали).  
* **DELETE /api/requests/:id** — Удаление ошибочно созданных записей.

## **3\. Компоненты UI (Frontend)**

### **3.1. Кнопка создания (AddRequestButton)**

Кнопка, которая всегда под рукой.

### **3.2. Форма создания/редактирования (RequestFormModal)**

Так как админ говорит по телефону, форма должна позволять заполнять данные «на лету»:

* **Быстрые поля:** Имя, Телефон, Тип события (Select).  
* **DateRangePicker:** Удобный выбор даты и диапазона времени (сразу заполняет start\_date и end\_date).  
* **Поле «Комментарий»:** Большое поле, куда админ вносит ключевые тезисы разговора.

### **3.3. Сетка карточек (RequestsGrid)**

Вместо строк таблицы — сетка (Grid).

**Компонент RequestCard:**

* **Акцент на Тип:** Яркая плашка с названием типа праздника.  
* **Тайминг:** Крупно дата и время начала.  
* **Клиент:** Имя и кликабельный номер телефона (чтобы можно было набрать в один клик с мобильной версии или через софтфон).  
* **Бюджет:** Выделенная сумма в углу карточки.

## **4\. Сценарий использования (Workflow)**

1. **Входящий звонок:** Админ берет трубку и параллельно нажимает **"+" (Создать заявку)** в админке.  
2. **Заполнение:** В процессе разговора он вбивает имя, выбирает в выпадашке "Детский праздник", кликает в календаре на субботу.  
3. **Сохранение:** Нажимает "Создать". Карточка мгновенно появляется в сетке.  
4. **Обработка:** Если по итогу звонка праздник подтвержден, админ прямо на карточке меняет статус на confirmed. Если нужно подумать — оставляет в new.  
5. **Контроль:** Заходя в админку утром, админ видит сетку карточек. Те, что горят цветом статуса new — требуют повторного звонка.
